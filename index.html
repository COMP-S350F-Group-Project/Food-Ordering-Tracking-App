Requirements Engineering

Vision & Scope

Our vision: a seamless city‑wide ordering & delivery experience that brings local restaurants closer to customers.

Version 1 focuses on five core capabilities: browse menus, customize items, place orders, pay, and track deliveries — within a 10 km radius.

User Roles

Non‑Functional Requirements

  • 99.9% monthly availability
  • End‑to‑end TLS, robust authz/authn, minimal PII collection

Speaker Notes (≈3:00):

We started with a clear vision: a seamless city‑wide ordering and delivery experience. To keep v1 realistic, we focused on five capabilities—browse menus, customize items, place orders, pay, and track delivery—within a 10 km radius.

We translated personas into user stories and prioritized with MoSCoW. Our must‑haves form the core: create orders with clear customizations; pay through a secure gateway; update order status reliably; stream courier location to customers; and push timely notifications.

To ensure design serves the experience, we set quantitative NFRs: P95 API latency under 250 ms, real‑time updates reach the client within 2 seconds, and 99.9% monthly availability. For security, we require end‑to‑end TLS, strong authz/authn, and minimal PII collection.

Speaker Notes (≈2:00):

Our architecture balances delivery speed and reliability. Cross‑platform mobile clients (e.g., Flutter or React Native) talk to a lightweight Backend for Frontend that aggregates calls to domain services. Core services include Users, Catalog, Orders, Delivery, Payments, and Notifications. We persist state in PostgreSQL with read replicas, use Redis for caching and rate limiting, and emit domain events to Kafka to decouple workflows and absorb spikes.

Real‑time tracking uses WebSockets so customers can subscribe to order events and courier location streams. We integrate a maps provider for routing and ETAs, and a payment gateway for authorization, capture, and refunds. End‑to‑end, we have tracing, metrics, and structured logging so we can hold SLOs and accelerate incident response.

Orders move through a strict state machine: Created → Confirmed → In Prep → Picked Up → Out for Delivery → Delivered → Completed, with Cancelled and Refunded branches where appropriate. Each transition is idempotent and auditable, and emits events for downstream consumers like notifications, analytics, and dashboards.

Speaker Notes (≈2:00):

We move fast and safely by leaning on process discipline. We use trunk‑based development with short‑lived PRs and feature flags so changes can merge early but roll out gradually. Interfaces are contract‑first, allowing clients and servers to evolve independently without surprises.

For resilience, we use idempotency keys, retries with backoff, and dead‑letter queues in payments and dispatch workflows. In CI/CD, we enforce quality gates: unit test coverage over 70%, contract checks, and latency smoke tests for key endpoints and WebSocket updates. Deployments are canary/blue‑green with automatic rollbacks, letting us observe real traffic safely and keep MTTR low.

We validate across the testing pyramid. Unit tests cover pricing logic, order transitions, and location parsing. Integration tests exercise the full path from order to payment to dispatch, and end‑to‑end tests confirm on real devices that customers can place orders and see live courier updates without manual intervention. We also run chaos experiments to simulate third‑party outages—maps or payments—and verify graceful degradation and recovery.

Looking ahead, Version 2 emphasizes customer‑visible value—coupons, group orders, and additional payment options—while we keep strengthening observability and security. Version 3 focuses on scale: multi‑city routing optimization and smarter dispatch algorithms. Throughout, one thing stays constant: process isn’t paperwork; it’s the system that enables us to deliver quickly, safely, and repeatably.
